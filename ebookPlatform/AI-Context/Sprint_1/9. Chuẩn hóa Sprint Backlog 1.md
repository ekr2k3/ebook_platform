# Task chuẩn hóa Sprint 1

## 1. Nhóm C1 – Upload ebook (8 SP)

### C1-01. Tạo giao diện form upload ebook

- **Mục tiêu:** Tạo trang Thymeleaf cho seller nhập thông tin ebook và chọn file upload.
    
- **Input:** title, author, description, file.
    
- **Output:** trang `upload.html` hiển thị form đầy đủ.
    
- **Điều kiện hoàn thành:**
    
    - Có các field: tên sách, tác giả, mô tả, file.
        
    - Có nút Upload.
        
    - Có vùng hiển thị lỗi/thành công.
        
- **Phụ thuộc:** không.
    
- **Ưu tiên:** cao.
    

### C1-02. Tạo `EbookController` cho màn hình upload

- **Mục tiêu:** Xây dựng controller để hiển thị form upload và nhận submit từ UI.
    
- **Input:** request GET/POST từ seller.
    
- **Output:** endpoint render form và nhận dữ liệu upload.
    
- **Điều kiện hoàn thành:**
    
    - Có endpoint GET hiển thị form upload.
        
    - Có endpoint POST nhận form submit.
        
    - Điều hướng đúng khi thành công/thất bại.
        
- **Phụ thuộc:** C1-01.
    

### C1-03. Xử lý upload file bằng `MultipartFile`

- **Mục tiêu:** Nhận file ebook từ request và đưa vào luồng xử lý nghiệp vụ.
    
- **Input:** file từ form upload.
    
- **Output:** file được đọc và chuyển tới service lưu trữ.
    
- **Điều kiện hoàn thành:**
    
    - Nhận được file từ request.
        
    - Kiểm tra file có tồn tại.
        
    - Không làm business logic trực tiếp trong controller.
        
- **Phụ thuộc:** C1-02.
    

### C1-04. Lưu file ebook lên Google Drive

- **Mục tiêu:** Upload file hợp lệ lên Google Drive.
    
- **Input:** file ebook hợp lệ.
    
- **Output:** link/id file trên Google Drive.
    
- **Điều kiện hoàn thành:**
    
    - Upload thành công trả về định danh file.
        
    - Có xử lý lỗi khi upload thất bại.
        
    - Không expose public link trực tiếp cho người dùng.
        
- **Phụ thuộc:** C1-03.
    

### C1-05. Lưu metadata ebook vào MySQL

- **Mục tiêu:** Lưu thông tin ebook sau khi upload thành công.
    
- **Input:** title, author, description, format, file size, drive file id/url, seller id.
    
- **Output:** bản ghi ebook trong DB.
    
- **Điều kiện hoàn thành:**
    
    - Lưu được metadata đầy đủ.
        
    - Có liên kết với seller.
        
    - Có trạng thái ban đầu phù hợp.
        
- **Phụ thuộc:** C1-04.
    

### C1-06. Validate kích thước file upload

- **Mục tiêu:** Chặn file vượt dung lượng cho phép trước khi lưu.
    
- **Input:** file upload.
    
- **Output:** kết quả hợp lệ/không hợp lệ.
    
- **Điều kiện hoàn thành:**
    
    - Có giới hạn dung lượng cấu hình được.
        
    - Trả message rõ ràng ra UI khi vượt quá giới hạn.
        
- **Phụ thuộc:** C1-03.
    

### C1-07. Hiển thị trạng thái upload ra giao diện

- **Mục tiêu:** Thông báo cho người dùng kết quả upload.
    
- **Input:** kết quả xử lý từ backend.
    
- **Output:** thông báo thành công/thất bại trên UI.
    
- **Điều kiện hoàn thành:**
    
    - Hiển thị message thành công.
        
    - Hiển thị message lỗi theo từng trường hợp.
        
- **Phụ thuộc:** C1-04, C1-05, C1-06.

## 2. Nhóm C2 – Validate file (5 SP)

### C2-01. Kiểm tra định dạng file PDF/EPUB

- **Mục tiêu:** Chỉ cho phép file đúng định dạng.
    
- **Input:** tên file, extension, content type.
    
- **Output:** valid/invalid.
    
- **Điều kiện hoàn thành:**
    
    - Chỉ chấp nhận PDF, EPUB.
        
    - File sai định dạng bị từ chối.
        
    - Có message lỗi rõ ràng.
        
- **Phụ thuộc:** C1-03.
    

### C2-02. Đọc file thử để validate cơ bản

- **Mục tiêu:** Kiểm tra file có thể mở/đọc ở mức cơ bản.
    
- **Input:** file PDF/EPUB đã qua bước check format.
    
- **Output:** valid/invalid.
    
- **Điều kiện hoàn thành:**
    
    - PDF đọc thử được.
        
    - EPUB đọc thử được ở mức đơn giản.
        
    - File hỏng bị đánh dấu lỗi.
        
- **Phụ thuộc:** C2-01.
    

### C2-03. Xử lý luồng file lỗi

- **Mục tiêu:** Khi file không hợp lệ, hệ thống dừng đúng chỗ và không lưu dữ liệu sai.
    
- **Input:** kết quả invalid từ validate.
    
- **Output:** request bị từ chối an toàn.
    
- **Điều kiện hoàn thành:**
    
    - Không upload lên Drive nếu file lỗi.
        
    - Không lưu metadata nếu file lỗi.
        
    - Có log lỗi hoặc trace xử lý.
        
- **Phụ thuộc:** C2-02.
    

### C2-04. Trả message lỗi validate ra UI

- **Mục tiêu:** Người dùng biết chính xác file lỗi vì lý do gì.
    
- **Input:** lỗi validate.
    
- **Output:** message lỗi trên giao diện upload.
    
- **Điều kiện hoàn thành:**
    
    - Có message cho sai format.
        
    - Có message cho file hỏng.
        
    - Có message cho file quá dung lượng.
        
- **Phụ thuộc:** C2-03.

## 3. Nhóm C3 – Preview ebook (8 SP)

### C3-01. Tạo endpoint preview ebook

- **Mục tiêu:** Cung cấp endpoint để xem thử ebook.
    
- **Input:** ebook id.
    
- **Output:** trang preview hoặc dữ liệu preview.
    
- **Điều kiện hoàn thành:**
    
    - Có endpoint preview.
        
    - Kiểm tra ebook tồn tại.
        
    - Kiểm tra trạng thái ebook cho phép preview.
        
- **Phụ thuộc:** C1-05.
    

### C3-02. Đọc file ebook từ Google Drive

- **Mục tiêu:** Lấy file từ Drive thông qua backend để phục vụ preview.
    
- **Input:** drive file id/url nội bộ.
    
- **Output:** stream/file content cho preview.
    
- **Điều kiện hoàn thành:**
    
    - Backend đọc được file.
        
    - Không trả public URL trực tiếp.
        
    - Có xử lý lỗi khi file không tồn tại trên Drive.
        
- **Phụ thuộc:** C3-01.
    

### C3-03. Render preview cho PDF

- **Mục tiêu:** Hiển thị preview PDF bằng iframe/viewer.
    
- **Input:** dữ liệu PDF hợp lệ.
    
- **Output:** màn hình preview PDF.
    
- **Điều kiện hoàn thành:**
    
    - PDF hiển thị được trên UI.
        
    - Không hiện nút download trực tiếp nếu viewer hỗ trợ.
        
- **Phụ thuộc:** C3-02.
    

### C3-04. Render preview đơn giản cho EPUB

- **Mục tiêu:** Hỗ trợ xem thử EPUB ở mức cơ bản trong Sprint 1.
    
- **Input:** dữ liệu EPUB hợp lệ.
    
- **Output:** màn hình preview EPUB tối giản.
    
- **Điều kiện hoàn thành:**
    
    - Có phương án hiển thị nội dung EPUB cơ bản.
        
    - Nếu chưa support đầy đủ thì phải có giới hạn/phạm vi rõ ràng.
        
- **Phụ thuộc:** C3-02.
    

### C3-05. Xây dựng UI preview bằng Thymeleaf

- **Mục tiêu:** Tạo trang preview hiển thị viewer và thông tin giới hạn.
    
- **Input:** ebook info + preview content.
    
- **Output:** `preview.html`.
    
- **Điều kiện hoàn thành:**
    
    - Có tên sách.
        
    - Có vùng hiển thị nội dung.
        
    - Có thông báo đây là bản xem thử.
        
- **Phụ thuộc:** C3-03, C3-04.
    

### C3-06. Giới hạn số trang/nội dung hiển thị

- **Mục tiêu:** Chỉ cho phép xem trước một phần ebook.
    
- **Input:** tổng số trang hoặc cấu trúc nội dung ebook.
    
- **Output:** nội dung preview giới hạn.
    
- **Điều kiện hoàn thành:**
    
    - PDF chỉ hiển thị số trang đầu theo rule.
        
    - EPUB chỉ hiển thị phần đầu theo rule đơn giản.
        
- **Phụ thuộc:** C3-03, C3-04.

## 4. Nhóm R5 – Preview cho buyer (5 SP)

### R5-01. Tạo trang chi tiết ebook

- **Mục tiêu:** Hiển thị thông tin ebook cho buyer trước khi bấm xem thử.
    
- **Input:** ebook id.
    
- **Output:** trang `detail.html`.
    
- **Điều kiện hoàn thành:**
    
    - Có tên sách, tác giả, mô tả, định dạng.
        
    - Có nút “Xem thử”.
        
- **Phụ thuộc:** C1-05.
    

### R5-02. Tạo hành vi nút “Xem thử”

- **Mục tiêu:** Buyer từ trang chi tiết có thể vào trang preview.
    
- **Input:** click action từ UI.
    
- **Output:** điều hướng tới endpoint preview.
    
- **Điều kiện hoàn thành:**
    
    - Nút hoạt động đúng.
        
    - Điều hướng đúng ebook.
        
- **Phụ thuộc:** R5-01, C3-01.
    

### R5-03. Áp dụng giới hạn preview cho buyer 5–10%

- **Mục tiêu:** Buyer chưa mua chỉ xem được một phần nhỏ nội dung.
    
- **Input:** role user + trạng thái sở hữu ebook.
    
- **Output:** rule preview cho buyer.
    
- **Điều kiện hoàn thành:**
    
    - Buyer chưa mua chỉ xem 5–10%.
        
    - Seller hoặc owner không dùng cùng rule này.
        
- **Phụ thuộc:** C3-06.
    

### R5-04. Không cho download trực tiếp file preview

- **Mục tiêu:** Giảm rủi ro người dùng tải file gốc.
    
- **Input:** luồng preview.
    
- **Output:** preview qua backend/viewer, không lộ direct link.
    
- **Điều kiện hoàn thành:**
    
    - Không render link Google Drive public.
        
    - Không có endpoint download công khai cho buyer preview.
        
- **Phụ thuộc:** C3-02, C3-05.

- **Điều kiện hoàn thành:**

- **Phụ thuộc:** R4-03.

# 6. Gợi ý format quản lý task trong Jira/Trello

Mỗi task nên có chung cấu trúc:

- **Task ID**
    
- **Tên task**
    
- **Module**
    
- **Mô tả**
    
- **Input**
    
- **Output**
    
- **Acceptance Criteria**
    
- **Phụ thuộc**
    
- **Assignee**
    
- **Estimate**
    
- **Trạng thái**
    

Ví dụ chuẩn:

**Task ID:** C1-04  
**Tên task:** Lưu file ebook lên Google Drive  
**Module:** Upload ebook  
**Mô tả:** Nhận file hợp lệ từ service upload và lưu lên Google Drive, trả về file id để lưu metadata.  
**Acceptance Criteria:**

- Upload thành công lên Drive
    
- Có xử lý exception
    
- Không lộ public URL trực tiếp
