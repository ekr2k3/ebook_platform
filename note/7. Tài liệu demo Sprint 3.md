# 🔧 5. Chi tiết User Story và kỹ thuật

---

## 🔵 US-S03 – Seller chọn cơ chế bảo vệ ebook

### Story

Là một Seller, tôi muốn chọn cơ chế bảo vệ ebook, để giảm nguy cơ sao chép trái phép.

### Acceptance Criteria

- Given tôi đang tạo hoặc chỉnh sửa ebook
    
- When tôi chọn chế độ bảo vệ
    
- Then hệ thống lưu cấu hình bảo vệ cho ebook đó
    

### Protection mode gợi ý

- `NONE`
    
- `WATERMARK`
    
- `RESTRICTED_ACCESS`
    

### Task kỹ thuật

- thêm field `protection_type` vào bảng `ebooks`
    
- thêm UI chọn protection type trong form seller
    
- validate giá trị hợp lệ
    
- hiển thị thông tin protection trên detail page
    

### SP

5

---

## 🔴 US-A05 – Admin cấu hình cơ chế bảo vệ

### Story

Là một Admin, tôi muốn cấu hình cơ chế bảo vệ ebook mặc định, để chuẩn hóa cách hệ thống bảo vệ nội dung.

### Acceptance Criteria

- Given admin truy cập trang cấu hình
    
- When chọn cấu hình mặc định
    
- Then hệ thống áp dụng cho ebook mới nếu seller không chọn riêng
    

### Task kỹ thuật

- tạo bảng `system_settings` hoặc config table
    
- CRUD config mặc định:
    
    - default protection type
        
    - enable sample preview
        
    - watermark text pattern
        
- service đọc cấu hình khi tạo ebook
    

### SP

5

---

## 🟢 US-B03 – Buyer xem preview ebook

### Story

Là một Buyer, tôi muốn xem trước một phần ebook, để đánh giá trước khi mua.

### Acceptance Criteria

- Given ebook đã được duyệt và có preview
    
- When buyer nhấn “Xem thử”
    
- Then hệ thống hiển thị sample nội dung giới hạn
    

### Task kỹ thuật

- thêm field `sample_file_url` hoặc rule sample pages
    
- tạo endpoint preview public
    
- chỉ cho xem số trang / chương giới hạn
    
- nếu protected thì preview không full content
    

### Cách làm thực tế

- PDF: cắt trước vài trang sample
    
- EPUB: sinh sample chapter đơn giản
    

### SP

5

---

## 🔵 US-S04 – Seller preview ebook trước publish

### Story

Là một Seller, tôi muốn xem trước ebook trước khi gửi duyệt, để chắc chắn file hiển thị đúng.

### Acceptance Criteria

- Given seller đã upload ebook
    
- When seller bấm preview
    
- Then hệ thống hiển thị nội dung ebook giống buyer sẽ thấy
    

### Task kỹ thuật

- route preview private cho seller
    
- chỉ seller sở hữu ebook mới xem được
    
- render nội dung preview từ file đã upload
    

### SP

3

---

## 🔵 US-S05 – Seller xem dashboard doanh thu

### Story

Là một Seller, tôi muốn xem tổng doanh thu và số lượt bán ebook, để theo dõi hiệu quả kinh doanh.

### Acceptance Criteria

- Given seller đăng nhập
    
- When mở dashboard
    
- Then hiển thị tổng số ebook, tổng đơn hàng, tổng doanh thu
    

### Chỉ số cơ bản

- số ebook đã đăng
    
- số ebook đã duyệt
    
- số lượt bán
    
- tổng doanh thu
    
- top ebook bán chạy
    

### Task kỹ thuật

- query từ `orders`, `order_items`, `ebooks`
    
- chỉ tính order `SUCCESS`
    
- render dashboard seller bằng Thymeleaf
    

### SP

5

## 🔴 US-A01 – Admin quản lý người dùng

### Story

Là một Admin, tôi muốn quản lý tài khoản người dùng, để kiểm soát hoạt động hệ thống.

### Acceptance Criteria

- Given admin truy cập trang user management
    
- When chọn một user
    
- Then có thể xem thông tin và thay đổi trạng thái active/inactive
    

### Task kỹ thuật

- trang danh sách user
    
- filter theo role, email, status
    
- action:
    
    - block
        
    - unblock
        
- tránh admin tự khóa chính mình
    

### SP

5

---

## 🔴 US-A04 – Admin xử lý ebook vi phạm

### Story

Là một Admin, tôi muốn ẩn hoặc gỡ ebook vi phạm, để bảo vệ hệ thống và tuân thủ chính sách nội dung.

### Acceptance Criteria

- Given có ebook vi phạm
    
- When admin thực hiện takedown
    
- Then ebook không còn hiển thị công khai và không thể mua mới
    

### Trạng thái gợi ý

- `PENDING`
    
- `APPROVED`
    
- `REJECTED`
    
- `TAKEDOWN`
    

### Task kỹ thuật

- thêm trạng thái `TAKEDOWN`
    
- admin action takedown
    
- catalog chỉ hiển thị `APPROVED`
    
- nếu ebook đã bị takedown:
    
    - không hiện trong search
        
    - không cho mua mới
        
    - có thể quyết định user cũ còn đọc được hay không
        

### SP

3

## ⚙️ OPT-01 – Tối ưu search + phân trang

### Story

Là một Buyer, tôi muốn kết quả tìm kiếm hiển thị nhanh và có phân trang, để dễ sử dụng khi số lượng ebook tăng.

### Acceptance Criteria

- Given có nhiều ebook
    
- When buyer tìm kiếm
    
- Then kết quả trả về theo trang, có sắp xếp rõ ràng
    

### Task kỹ thuật

- dùng `Pageable` trong Spring Data
    
- thêm index cho:
    
    - title
        
    - author
        
    - status
        
- query chỉ lấy sách approved
    
- UI phân trang
    

### SP

3

---

## ⚙️ OPT-02 – Tối ưu SSR caching cơ bản

### Story

Là một Admin, tôi muốn áp dụng chính sách cache phù hợp cho trang public và private, để tăng tốc hệ thống mà không rò rỉ dữ liệu người dùng.

### Acceptance Criteria

- Given người dùng truy cập trang công khai
    
- When tải lại page
    
- Then response có thể cache phù hợp
    
- Given người dùng truy cập thư viện hoặc dashboard
    
- When tải page
    
- Then response không được cache sai
    

### Task kỹ thuật

- set cache header cho page public
    
- `no-store` cho page private
    
- tách rõ:
    
    - public catalog
        
    - private library/dashboard
        
- kiểm tra không lộ dữ liệu session
    

### SP

3

# ⚠️ 11. Rủi ro

---

## 11.1 Preview sinh không đúng nội dung

**Rủi ro**

- sample file lỗi
    
- preview lộ quá nhiều nội dung
    

**Giải pháp**

- giới hạn số trang/chương cố định
    
- render sample riêng, không dùng full file
    

---

## 11.2 Watermark không hiệu quả

**Rủi ro**

- watermark không xuất hiện
    
- người dùng vẫn tải bản sạch
    

**Giải pháp**

- watermark áp dụng ở layer stream/download
    
- kiểm tra đường dẫn file private
    

---

## 11.3 Dashboard query chậm

**Rủi ro**

- join nhiều bảng làm chậm seller dashboard
    

**Giải pháp**

- query summary riêng
    
- thêm index
    
- chỉ thống kê dữ liệu cần thiết
    

---

## 11.4 Takedown ảnh hưởng user cũ

**Rủi ro**

- ebook bị gỡ khiến buyer cũ không đọc được, gây tranh chấp
    

**Giải pháp**

- định nghĩa rule rõ:
    
    - takedown = ngừng bán mới
        
    - hay takedown = chặn toàn bộ truy cập
        
- rule này cần được thống nhất với nghiệp vụ
    

---

## 11.5 Cache SSR sai

**Rủi ro**

- page private bị cache
    
- library/dashboard hiển thị sai user
    

**Giải pháp**

- `Cache-Control: no-store` cho private page
    
- test bằng nhiều user/session
